Cairn.

An agent-native framework for distributed research collaboration — where humans and AI contribute to a shared, durable project memory.

SubstrateGit repository

§1 — What is Cairn

A shared collaborator for the research group.

Cairn is a tool and design standard that lets a research group's collective thinking accumulate over time, become queryable, and remain visible across asynchronous work and changing personnel. It defines repository conventions, file schemas, and patterns for collaboration, and provides a Python package for scaffolding and managing project instances. It supports parallel lines of inquiry through branching and treats human and AI contributors uniformly within the same workflow.

The name borrows from the stone markers travelers build along trails. Each person who passes adds to the pile. The cairn persists for those who follow, points to the path, and survives the absence of any individual builder. Its construction is collective; its inspection is open; its purpose is to make the group's accumulated knowledge available to anyone walking the route, including the people who will come later.

The word carries over: a research project's repository, organized following Cairn's conventions, is itself a cairn. There is one cairn per project. Cairn — capitalized — is the framework that defines what a cairn looks like and provides the tooling to create one.

Research groups already have collective knowledge. It lives in meetings, side conversations, individual memories, scattered documents, and the slow accretion of folklore among long-tenured members. Cairn is an attempt to make that substrate explicit, durable, and shared — without burdening the group with overhead that the AI itself can absorb.

What it enables

A collaborator who missed Tuesday's meeting can ask the agent what was discussed, what decisions were made, and whether their concern was raised. A new postdoc joining six months in can ask why the project went one direction rather than another and get an answer with citations to the meetings where it was decided. A student exploring an alternative approach can do so on a branch, without disrupting the canonical direction, and can later propose merging their finding back. An AI collaborator can monitor recent literature in the background and surface relevant work without anyone having to ask.

What ties all of these together is a single architectural commitment: the group's shared memory lives in one inspectable, versioned place, and every component — human or AI, synchronous or async — interacts with it through a uniform interface.

§2 — Principles

Five commitments that shape every design choice.

01

The repository is the source of truth.

All canonical project state lives in a git repository. Every other component is a derived view or interface layer.

02

Derived systems are disposable.

Vector indices, MCP servers, dashboards — all replaceable without compromising the substrate. The project survives them.

03

Uniform collaborator abstraction.

Human and AI collaborators use the same interfaces. Identity differs; mechanism does not.

04

Branches as first-class.

Parallel exploration is encouraged. Merges are explicit, reviewed events that themselves become artifacts.

05

Layered consumption, layered distribution.

Skills replicate locally. Knowledge and state are read via tools. Each layer is distributed appropriately for its access pattern.

§3 — Architecture

The simplest version is the most important.

Cairn's architecture is layered so that each layer earns its complexity. The base — a cairn itself — is a git repository organized following Cairn's conventions. Everything else is optional augmentation that can be added when the underlying friction is felt, and removed without breaking what came before. This is the architectural commitment that gives the system its durability.

3.1 — Git-native, by itself

In the simplest configuration, a cairn is the entire system — just a git repository. Each collaborator clones it locally. Their Claude Code session reads the repo from disk: a small orientation file tells the agent where things live; from there it loads what it needs. Meeting transcripts, decisions, findings, literature notes, and provenance artifacts (model cards, RO-Crates, AI usage records) all sit as plain files. Writes are normal git operations — commits, pull requests, code review. Permissions and history are GitHub's.

This is enough for a surprisingly long time. A small research group with a year of meeting notes can run on nothing but this and find the agent extraordinarily capable, because Claude is good at navigating well-organized text. No services to run; nothing to maintain.

3.2 — With MCP, when needed

As the project grows, two kinds of friction emerge. Searching across many meeting notes becomes slow. Semantic queries — "find prior discussions related to this new finding" — start needing more than direct file reads. The response is to add an MCP server alongside the repo: it reads from the repo, maintains a derived vector index, and exposes curated semantic tools to client agents. Tool names are chosen for the lab's actual queries (get_decisions_about, find_related_prior_discussion) rather than as generic search.

Critically, this is additive. The repo remains the source of truth. The MCP server can be added, replaced, or removed without altering the substrate.

3.3 — The full ecology of clients

Once a core exists, many kinds of clients can interact with it. Human collaborators with Claude Code on their laptops. AI collaborators like a literature monitor running on a schedule. A Slack bot for chat-based queries from a phone. A web dashboard visualizing active branches. A meeting capture module ingesting audio and submitting structured transcripts.

The architecture is intentionally agnostic about which clients exist. New ones are added by writing modules that follow the same interface conventions. Nothing in the substrate has to change to accommodate them.

§4 — Branching

Parallel thinking, made legible.

A research group's work is rarely linear. Different members pursue different angles. Some ideas get explored at length before being set aside. Some explorations turn into the main direction. The system should support this naturally — not by hiding it, but by making it explicit.

Cairn uses git branches as first-class objects. A collaborator (human or AI) creates a branch when they want to explore a line of inquiry. Their findings, proposed decisions, and exploratory work live there without polluting canonical state. The knowledge base — meeting notes, prior findings — is shared across branches; only the interpretive and forward-looking layer diverges. When a branch's work is ready to merge, it goes through a pull request: review, discussion, accept or revise.

Branches that don't merge aren't failures. An exploratory branch that didn't pan out is itself useful history — a record of what was tried and why it was set aside. The graph of branches over time is a record of how the group reasoned through the project.

§5 — Reproducible Artifacts

Documenting the provenance of AI-assisted research.

A cairn captures a project's living state. When findings are ready to be shared externally — a paper with reproducible code, a dataset release, a benchmark submission — they get exported as reproducible research artifacts that document how the work was done, what data and methods were used, and which agents contributed. With AI agents increasingly involved in research, this documentation is becoming non-negotiable: journals, funders, and downstream readers all need to know what AI did and didn't do.

The standards for this are still emerging and competing. Cairn provides a directory (knowledge/provenance/) where these artifacts live, but defers to existing standards for their internal structure. Two stand out as directly aligned with Cairn's commitments. ASTRA (Agentic Schema for Transparent Research and Analysis) — Lanusse & Parker, 2026, Lightcone Research — is an open specification under active development that articulates three properties an analysis must satisfy to be efficiently vetted: provenance-certified, fully observable, and scientifically legible. Like Cairn, it commits to being a specification rather than a platform. ARA (Agent-Native Research Artifact) by Liu et al. structures research with cross-layer bindings between claims, code, exploration graph, and evidence, and uses provenance tags to distinguish human-confirmed facts from AI inferences. Broader foundations include RO-Crate for JSON-LD packaging, W3C PROV for foundational vocabulary, and workflow systems like Snakemake, Nextflow, and REANA for reproducible computational pipelines.

The separation matters. A cairn's internal structure is optimized for the group's day-to-day collaborators navigating evolving state. A reproducible artifact is crystallized for outside readers and downstream agents trying to reproduce or extend the work. Both rest on the same bet: the substrate should be a specification rather than a platform, so that an ecosystem of tools can grow around it. The transformation between them is itself a meaningful step — where the project decides what part of its accumulated thinking gets fixed for external use.

§6 — Build Path

Each phase earns its complexity.

A persistent risk in projects like this is building infrastructure for problems that haven't been encountered yet. The build path is designed to defer every piece of complexity until friction warrants it — and to make each addition cheap because the layer beneath stays unchanged.

0week 1-2

Discovery

Build the repository structure by hand for one real project. Use it for actual meetings. Refine the schema as friction appears.

1month 2-3

Template

Once the structure is stable, convert to a cookiecutter template for spinning up new projects with one command.

2month 4+

Python package

A thin cairn package providing schema validation, parsers, helpers, and an init command. Codifies what's stable; resists feature creep.

3—

MCP server

A centralized server reading from the repo, maintaining a vector index, exposing semantic tools. Sessions can query the project from anywhere.

4—

Meeting automation

Streaming Whisper transcription, pyannote diarization, voice-identification across meetings. Live transcripts during meetings; speaker-attributed commits within minutes of ending.

5—

AI collaborators

First addition: a literature monitor with its own branch. Evaluate. Add more as warranted.

6long-term

Voice mode

Real-time voice participant in meetings, using the same Cairn interface as everything else. Speaks when called on; logs to the same store.

§7 — Design Decisions

Choices worth recording.

Several decisions shape the system in ways that aren't obvious from the components alone. Recording them here makes future revision deliberate rather than accidental.

Why git, not a database.

Git already provides versioning, branching, merging, attribution, distributed access, conflict detection, review workflows, and offline capability. Building these on top of a database would be reinventing what exists, less well. A research group's project memory has no requirements that git can't meet.

Why YAML, not JSON.

YAML is human-editable. Researchers will sometimes want to edit state files by hand, and YAML's syntax is friendlier for that. The primary use case is human-readable structured state, not tooling.

Why MCP is optional, not foundational.

A research group with a year of history can run entirely on git-based access. Making MCP optional preserves the survivability of the system: if the MCP ecosystem changes or the server breaks, the project continues working.

Why each AI collaborator has its own identity.

Distinct identities let the group evaluate AI contributions on their own terms, attribute work to specific automated processes, and configure permissions per-agent. It matches the uniform-collaborator principle: AI collaborators are participants, not invocations of a generic tool.

Why a review queue for AI writes.

AI contributions can be valuable and can also be noise. A review queue creates a deliberate moment where humans evaluate what to promote, which both protects canonical state and provides feedback signal for tuning AI collaborators over time.

Why per-project repositories, not a single monorepo.

Projects have their own lifecycles, their own collaborators, and their own access boundaries. Per-project repositories match this. Cross-project capability — a collaborator working on project A asking what's relevant from project B — is achieved by their session being pointed at multiple repos, not by collapsing the structure.
